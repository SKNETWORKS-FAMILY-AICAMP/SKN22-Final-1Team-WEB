# 로그아웃 후 뒤로가기 캐시 관련 체크포인트

작성일: 0401  
버전: ver1

## 1. 개요

현재 QA에서 `로그인 -> 로그아웃 -> 브라우저 뒤로가기` 순서로 이동할 경우,  
화면상으로는 여전히 로그인된 것처럼 보인다는 피드백이 확인되었습니다.

이 이슈는 프론트 단독 문제가 아니라,

- 브라우저의 `bfcache(back-forward cache)`
- 백엔드 응답의 캐시 금지 헤더
- 프론트의 뒤로가기 복원 시 재검증

이 함께 맞물리는 유형으로 판단됩니다.

## 2. 백엔드에서 반영한 사항

- 인증/민감 페이지 및 로그아웃 관련 뷰에 `never_cache` 적용
- 응답 헤더 기준 캐시 금지 처리 강화
- 공통 레이아웃에 `pageshow` 복원 감지 스크립트 추가
  - 뒤로가기로 복원된 페이지가 로그인 상태 화면이면 자동 새로고침 유도

적용 범위 예시:

- customer login / menu / survey / camera / result / history / trend
- partner login / dashboard / staff / designer management
- logout / verify / dashboard enter

## 3. 프론트 체크포인트

### 3-1. 로그아웃 직후 이동 방식

- [ ] 로그아웃 후 이동 시 `href` 단순 이동만 쓰는지 확인
- [ ] 가능하면 `location.replace(...)` 또는 동등한 방식으로 history 잔존 영향을 줄이는지 확인

### 3-2. 뒤로가기 복원 감지

- [ ] `pageshow` 이벤트에서 `event.persisted === true`를 감지하는지 확인
- [ ] `navigation.type === 'back_forward'` 케이스도 함께 고려하는지 확인
- [ ] 복원된 화면이 로그인 상태 UI일 경우 즉시 세션 재검증 또는 강제 새로고침하는지 확인

### 3-3. 프론트 상태 초기화

- [ ] 로그아웃 시 메모리 상태, 선택 상태, 로그인 표시 UI가 같이 초기화되는지 확인
- [ ] 세션 스토리지 / 로컬 스토리지에 로그인 상태를 따로 남기고 있지 않은지 확인
- [ ] 남기고 있다면 로그아웃 시점에 함께 정리되는지 확인

### 3-4. 보호 페이지 재진입 처리

- [ ] 뒤로가기로 `partner/dashboard`, `partner/staff`, `customer/menu`, `customer/result` 등에 복원 진입 시 실제 서버 응답 기준으로 다시 판정하는지 확인
- [ ] 화면만 남아 있고 API는 실패하는 반쯤 로그인 상태가 발생하지 않는지 확인

## 4. 프론트/백 공통 재현 체크

- [ ] customer 로그인 후 로그아웃 -> 뒤로가기 시 로그인 상태 화면이 그대로 남는지 확인
- [ ] partner(shop owner) 로그인 후 로그아웃 -> 뒤로가기 시 owner 대시보드가 그대로 보이는지 확인
- [ ] designer 로그인 후 로그아웃 -> 뒤로가기 시 staff 화면이 그대로 보이는지 확인
- [ ] 뒤로가기 후 새로고침 없이도 UI가 자동 복구/차단되는지 확인

## 5. 판단

이 이슈는 프론트만 수정해도 일부 완화는 가능하지만,
백엔드의 캐시 금지 헤더 지원이 없으면 브라우저별 편차가 크게 남을 수 있습니다.

반대로 백엔드만 수정하고 프론트에서 `pageshow` 복원 대응을 하지 않으면,
일부 환경에서는 여전히 “화면상 로그인 유지”처럼 보일 수 있습니다.

따라서 이번 건은

- 백엔드: 캐시 금지 헤더
- 프론트: 뒤로가기 복원 시 재검증

을 같이 적용하는 방식으로 보는 것이 가장 안전합니다.

## 6. 요청사항

- [ ] 프론트에서 로그아웃 후 뒤로가기 복원 시점의 상태 관리 로직을 점검
- [ ] `pageshow` 기반 재검증 또는 강제 새로고침 대응 적용 여부 확인
- [ ] 로컬/세션 스토리지에 로그인 상태를 별도 유지하는 코드가 있으면 정리
- [ ] customer / partner / designer 각 흐름별로 뒤로가기 재현 테스트 진행
