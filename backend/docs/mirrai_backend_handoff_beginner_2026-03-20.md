# MirrAI Backend 인수인계 메모

작성자 관점: veteran 팀장이 초보 팀원에게 설명하는 버전

## 1. 이번에 왜 바꿨는가

이번 정리의 핵심은 추천 화면을 3가지로 나누기 위해서였습니다.

- `기존 스타일`
- `트렌드`
- `이번 촬영 결과`

예전에는 이 셋의 의미가 코드 안에서 조금 섞여 있었습니다.
그래서 지금은 의미를 분명하게 나눴습니다.

## 2. 제일 중요한 개념 2개

### `former_recommendations`

이건 "예전에 저장된 추천 결과"입니다.

예를 들면:

- 예전에 추천받았던 스타일
- 예전에 내가 골랐던 스타일
- 트렌드에서 골라서 상담 보낸 스타일

이런 것들이 여기에 남습니다.

즉, "기존 스타일" 화면에서 보여줄 데이터입니다.

### `current recommendations`

이건 "이번 촬영 기준으로 새로 만든 5개 결과"입니다.

즉, 카메라 촬영 후 결과 화면에서 보여줄 데이터입니다.

둘을 분리한 이유는:

- 과거 이력과
- 방금 생성한 새 결과가

한 화면에서 섞이면 사용자가 헷갈리기 때문입니다.

## 3. 지금 추천은 어떻게 저장되나

새 촬영이 끝나면 backend가 추천 5개를 만들고, 그 5개를 DB에 저장합니다.

이 저장 테이블 이름이 `former_recommendations`입니다.

조금 헷갈릴 수 있는데, 이름은 former지만 실제로는:

- 막 생성된 결과도 일단 저장
- 시간이 지나면 그게 과거 이력이 됨

이렇게 이해하면 됩니다.

즉, "새 결과"와 "옛 결과"가 같은 테이블에 저장되지만,
API에서 어떤 것만 보여줄지 다르게 나누는 구조입니다.

## 4. 화면별로 어떤 API를 쓰면 되는가

### 1) 기존 스타일 화면

`GET /api/v1/analysis/former-recommendations/`

이 API는:

- 예전에 선택한 것부터 먼저 보여주고
- 5개가 안 되면 나머지 과거 추천을 채워서
- 최대 5개를 보여줍니다

### 2) 트렌드 화면

`GET /api/v1/analysis/trend/?days=30`

이 API는:

- 최근 30일 동안 많이 선택된 스타일을 보여줍니다

### 3) 이번 결과 화면

`GET /api/v1/analysis/recommendations/`

이 API는:

- 이번 촬영 기준으로 생성된 최신 5개를 보여줍니다

## 5. survey skip은 이제 어떻게 되나

예전에는 설문이 없으면 추천 API가 404가 나는 쪽에 가까웠습니다.

지금은 그렇게 하지 않고:

- 설문도 없고 촬영도 없으면
  - "설문을 하거나 촬영을 시작해주세요" 같은 안내를 줍니다
- 설문은 없지만 촬영은 있으면
  - 얼굴 분석만으로 추천 5개를 만듭니다

즉, 사용자 입장에서 막히는 느낌보다 "다음 행동"이 보이게 바뀌었습니다.

## 6. confirm을 누르면 무슨 일이 생기나

사용자가 스타일을 확정하면 backend는 3가지를 같이 처리합니다.

1. 어떤 추천을 골랐는지 저장
2. 스타일 선택 로그 저장
3. 관리자에게 상담 요청 로그 저장

이렇게 해두면:

- 나중에 `기존 스타일` 화면에서 다시 보여줄 수 있고
- 트렌드 집계에도 쓸 수 있고
- 상담 전송 이력도 남습니다

## 7. 카드 클릭 시 무엇을 프론트가 받나

추천 카드 하나 안에는 이제 이런 데이터가 들어 있습니다.

- `sample_image_url`
- `simulation_image_url`
- `llm_explanation`
- `style_name`
- `match_score`

그래서 사용자가 카드를 클릭하면,
같은 페이지 안의 다른 컴포넌트에서:

- 샘플 이미지
- 설명 문구

를 바로 보여줄 수 있습니다.

## 8. 하지만 아직 진짜 AI 생성이 끝난 것은 아니다

이 부분이 중요합니다.

지금 구조는 "흐름"과 "저장 방식"을 먼저 정리한 상태입니다.

아직 완전히 붙지 않은 것:

- 실제 MediaPipe 얼굴 분석
- 실제 Hair simulation 생성
- 실제 LLM 설명 생성

지금은 placeholder 또는 rule-based 추천입니다.

즉,
"추천 5개가 나온다"는 흐름은 맞지만,
"그 5개 이미지가 실제 생성 모델 산출물이다"까지는 아직 아닙니다.

## 9. main.py는 왜 남겼는가

이전에는 `main.py`가 애매했습니다.

- Django도 있고
- FastAPI도 있고
- 누가 중심인지 불분명했습니다

그래서 이번에는 `main.py` 역할을 분명하게 바꿨습니다.

이제 `main.py`는:

- 내부 AI 서비스
- 얼굴 분석
- 추천 5개 생성
- 설명 생성

같은 AI 쪽만 맡는 파일로 보는 게 맞습니다.

즉:

- Django = 사용자용 서비스
- main.py = AI 전용 서비스

이렇게 역할을 나눈 것입니다.

나중에 AI 서버를 따로 분리하고 싶을 때 이 구조가 도움이 됩니다.

## 10. requirements.txt는 지금 더 바꿔야 하나

현재 단계에서는 추가 변경이 꼭 필요하지 않습니다.

이유:

- `main.py` 유지에 필요한 FastAPI 관련 패키지는 이미 들어가 있음
- 이번 작업에서 새 외부 패키지를 더 도입하지 않았음

## 11. 초보 관점에서 기억하면 되는 한 줄

지금 backend는

- 과거 추천
- 트렌드
- 새 촬영 결과

이 3개를 구분해서 보여줄 수 있게 정리됐고,
사용자가 최종 선택하면 그 결과를 다시 이력과 상담 요청으로 남길 수 있게 바뀌었습니다.

다만 AI 품질 자체는 아직 후속 model 연동이 필요합니다.
