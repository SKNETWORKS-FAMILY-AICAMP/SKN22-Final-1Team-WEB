## 1. 목적

신규 이슈 `#184 추천/결과 UX 안정화`, `#185 프론트 기준 화면 정합성`에 착수할 때
현재 백엔드가 이미 보장하는 범위와, 백엔드에서 먼저 정리해둘 수 있는 작업을 분리하기 위한 문서입니다.

## 2. #184 추천/결과 UX 안정화

### 현재 백엔드가 보장하는 것

- 고객 흐름 기본 순서: `정보 입력 -> 촬영 -> 설문 -> 결과`
- 결과 페이지 라우트 존재: `/customer/result/`
- 촬영 성공 후 기본 이동은 설문으로 정리되어 있음
- 로컬/검증 환경에서는 `MIRRAI_LOCAL_MOCK_RESULTS=True`일 때 mock 결과 fallback 가능
- 결과 조회용 추천 데이터는 시드 고객 기준으로 Supabase에 적재되어 있음

### 백엔드에서 먼저 확인할 작업

- [ ] 결과 페이지에서 `실제 결과`와 `mock 결과`를 구분하는 응답 필드와 화면 표시 기준을 확정
- [ ] 추천이 아직 준비되지 않은 경우의 빈 상태 메시지/재시도 기준을 정리
- [ ] `촬영 완료 -> 설문 -> 결과` 중간 단계에서 이탈했을 때 재진입 규칙을 점검
- [ ] 결과 페이지가 참조하는 추천/분석 데이터 계약을 문서화

### 프론트와 함께 맞춰야 할 작업

- [ ] 로딩 상태 문구와 UX
- [ ] 결과 카드 노출 방식
- [ ] 추천 없음/지연 중 상태에서의 사용자 안내

## 3. #185 프론트 기준 화면 정합성

### 현재 백엔드가 보장하는 것

- shop owner와 designer 권한 분리
- owner 화면: `/partner/dashboard/`
- designer 화면: `/partner/staff/`
- shop 로그인 후 디자이너 선택 단계 존재
- owner만 고객 배정/재배정, 트렌드 리포트 접근 가능
- designer는 본인 고객만 조회 가능

### 백엔드에서 먼저 확인할 작업

- [ ] `partner` 관련 템플릿에서 owner/designer 조건 분기가 최신 프론트 UI와 충돌하지 않는지 정리
- [ ] 프론트가 의존하는 핵심 요소(`보안 가이드`, `트렌드 리포트`, 배정 UI, 챗봇)의 데이터/권한 조건을 표로 정리
- [ ] CTA와 이동 경로(`로그인`, `파트너 센터`, `매장 전체 대시보드로 이동`)를 다시 점검
- [ ] `/partner/`, `/partner/dashboard/`, `/partner/staff/` 기준 URL별 기대 동작을 문서화

### 프론트와 함께 맞춰야 할 작업

- [ ] 최종 기준 스크린샷/Figma 확보
- [ ] owner 화면과 designer 화면의 시각적 차이 확정
- [ ] 버튼 위치, 문구, 정보 카드 노출 방식 확정

## 4. 백엔드 단독 착수 우선순위

백엔드만으로도 바로 진행 가능한 순서는 아래와 같습니다.

1. `#184` 결과/빈 상태 데이터 계약 정리
2. `#185` URL별 권한/노출 조건 표 작성
3. 프론트 기준 화면 자료 수신 후 템플릿 정합성 수정

## 5. 비고

- `#184`는 추천 데이터와 결과 상태 처리 규칙이 핵심입니다.
- `#185`는 템플릿 자체보다 **권한에 따라 무엇이 보여야 하는지**를 먼저 명확히 하는 것이 중요합니다.
