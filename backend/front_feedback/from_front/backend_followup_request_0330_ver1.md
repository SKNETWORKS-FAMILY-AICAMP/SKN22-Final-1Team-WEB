# 백엔드 후속 요청서 - 프론트 확인용

## #32 회원가입 후 이동 흐름

- [x] 회원가입이 끝난 뒤 사용자가 바로 `partner/dashboard/`로 이동하는지 확인
- [x] 아직 로그인 단계로 되돌아가는 흐름이 남아 있지 않은지 확인
- [x] 사업자등록번호가 10자리 형식이 아닐 때 front에서 먼저 안내하거나 제출을 막는지 확인

설명:

이 항목은 회원가입 이후 화면 이동이 이미 합의된 흐름과 맞는지 확인하는 것입니다. 현재 backend는 회원가입 직후 대시보드로 이어지는 흐름을 기대하고 있습니다.
또한 최근 400 오류 분석 기준으로, 사업자등록번호는 backend 최종 검증 전에 front에서 자리수와 형식을 먼저 확인해주는 편이 더 안전합니다.

## #123 재추천 결과 갱신

- [x] 재추천 시 기존 5장 대신 새로운 5장이 전부 다시 출력되는지 확인
- [x] 일부 카드만 바뀌는 방식이 아니라 화면 전체가 새 결과로 다시 그려지는지 확인

설명:

합의된 방향은 부분 교체가 아니라 전면 교체입니다. backend는 재추천을 1회성, 취향 우선 재평가로 처리하고 있으므로 front에서는 실제 화면이 그 흐름대로 바뀌는지만 확인하면 됩니다.

## #140 공통 에러 응답 형태

- [x] backend가 `detail`, `message`, `error_code`를 함께 내려도 화면이 정상 동작하는지 확인
- [x] 검증 오류와 인증 오류가 현재 UI 흐름을 깨지 않는지 확인

설명:

backend는 에러 응답을 더 일관된 형식으로 보내도록 정리했습니다. front에서는 현재 화면이 이 응답을 무리 없이 받아들이는지만 확인하면 됩니다.

## #141 24시간 만료 후 로그아웃 흐름

- [x] 24시간 세션이 끝난 뒤 사용자가 자연스럽게 로그인 화면으로 돌아가는지 확인
- [x] 만료 상태가 사용자에게 헷갈리지 않게 보여지는지 확인

설명:

이 항목은 토큰 생성 방식이 아니라, 세션이 만료되었을 때 화면에서 어떤 흐름과 문구가 보이는지를 확인하는 작업입니다.

## 추가 응답 필드 확인

- [x] `storage_snapshot` 같은 추가 응답 필드가 화면을 깨지 않거나, 필요하면 안전하게 무시되는지 확인

설명:

backend는 디버깅과 추적을 위해 응답에 추가 정보를 넣고 있습니다. front는 이 값 때문에 화면이 깨지지 않는지만 확인하면 됩니다.
