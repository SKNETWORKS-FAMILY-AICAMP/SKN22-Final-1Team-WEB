# 백엔드가 front에 요청하는 확인 사항
---

## 어떤 내용을 요청하나요?

> backend에서 다음 단계로 넘어가기 전에, front 쪽에서 먼저 정리하거나 확인해주면 좋은 일을 모아둔 체크리스트입니다.
>
> 가능한 한 front가 직접 처리해야 할 일만 적었습니다.

## 작업 상세 내용

### 배포가 잘 뜨는지 먼저 확인

- [x] `gunicorn` 실행할 때 `--pythonpath /app`를 넣었는지 확인
- [x] `backend/app/apps.py`와 `settings.py`에서 앱 이름이 같은지 확인
- [x] `app`과 `App`처럼 대소문자가 섞여 있지 않은지 확인
- [x] `backend/app/__init__.py` 파일이 있고, Git에도 올라가 있는지 확인
- [x] Docker 캐시를 지운 뒤 다시 빌드했는지 확인

**설명**
- 이 부분은 기능보다 먼저 보는 항목입니다.
- 배포가 502 없이 떠야 다음 화면 확인과 API 확인을 할 수 있습니다.

### `#32` 매장 계정 생성 뒤 화면 이동 확인

- [x] 계정 생성이 끝나면 어디 화면으로 갈지 정하기
- [x] `partner/signup/`가 진짜 가입 화면으로 쓰이는지 확인
- [x] 가입이 끝나면 `partner/dashboard/`로 바로 가는지 확인
- [x] 가입이 끝난 뒤 back이 준 안내를 front에서 어떻게 받을지 정하기

**설명**
- 계정 생성 자체는 backend에서 준비되어 있습니다.
- 이제는 front에서 가입이 끝난 다음 어디로 갈지 정하면 됩니다.

### `#123` 재추천 화면 확인

- [x] 처음 추천 화면에서 `재생성` 버튼과 `미용사 상담` 버튼을 어디에 둘지 정하기
- [x] 재추천은 한 번만 된다는 점을 화면에서 어떻게 보여줄지 정하기
- [x] 재추천 뒤에는 `재생성` 버튼을 숨기고 `미용사 상담`만 남길지 확인
- [x] `recommendation_id`를 front에서 어떻게 기억할지 정하기
- [x] 재추천 뒤 카드 갱신을 다시 불러올지, 일부만 바꿀지 정하기

**설명**
- 처음 추천과 재추천은 같은 기능처럼 보여도 실제 규칙이 다릅니다.
- 그래서 버튼 위치와 다시 보여주는 방식부터 맞춰야 합니다.

### `#140` 에러 화면 확인

- [x] 지금 backend가 주는 기본 에러 형식을 front가 바로 받을 수 있는지 확인
- [x] 입력 오류와 일반 오류를 화면에서 어떻게 다르게 보여줄지 정하기
- [x] 나중에 공통 에러 형식이 들어오기 전까지 지금 방식으로 처리 가능한지 확인

**설명**
- 지금은 backend가 기본 에러 형식을 우선 쓰고 있습니다.
- front가 이 형식을 먼저 받아줄 수 있는지만 확인하면 됩니다.

### `#141` 로그인 유지 방식 확인

- [x] 로그인 정보와 다시 쓰는 정보를 어디에 저장할지 정하기
- [x] 로그인 후 24시간 유지라는 점을 front에서 어떻게 보여줄지 정하기
- [x] 로그인 시간이 끝났을 때 다시 로그인하게 만드는 흐름을 정하기
- [x] 다시 쓰는 정보가 실패했을 때 어떤 문구를 보여줄지 정하기

**설명**
- backend는 로그인 유지용 정보를 주고받는 기준을 반영했습니다.
- 남은 것은 front에서 저장 방식과 안내 문구를 맞추는 일입니다.
