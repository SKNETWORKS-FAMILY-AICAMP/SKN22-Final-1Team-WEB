# 프론트 후속 확인 요청서 `ver_3_1`

이 문서는 `#135`를 제외한 현재 이슈들 중,  
PM 답변이 나온 뒤에도 프론트에서 추가로 확인하거나 구현해야 하는 항목만 다시 추린 체크리스트입니다.

기준:
- PM이 이미 결정한 내용은 다시 묻지 않습니다.
- 백엔드가 이미 준비한 내용은 설명만 남기고, 프론트 확인이 필요한 것만 체크박스로 남깁니다.
- 초보 개발자와 GPT 에이전트가 함께 읽는 상황을 고려해, 가능한 한 바로 구현할 수 있는 질문으로 적었습니다.

## `#32` 매장 계정 생성

- 동의 항목 저장 흐름은 준비되어 있습니다.
- 가입 성공 응답은 줄 수 있습니다.

프론트 확인 체크리스트:
- [x] 가입 성공 후 `/partner/dashboard/`로 이동하도록 분기했는지 확인
- [x] `partner/signup/`가 실제 회원가입 화면으로 쓰이는지 최종 확인
- [x] 가입 성공 후 자동 로그인 상태로 대시보드에 진입하는 흐름인지 확인

왜 필요한가:
- PM이 첫 화면은 정해줬고, 이제 프론트에서 실제 라우팅과 화면 연결만 맞추면 됩니다.

## `#123` 추천 시뮬레이션 재생성 API 제공

PM 결정:
- 재생성 기능은 유지합니다.
- 같은 프롬프트를 쓰되 시드값을 바꾸는 방식으로 갑니다.
- 재생성은 `1회만` 허용합니다.
- 재생성 시에는 `trend` 비중을 낮추고, `얼굴 비율`과 `취향 벡터` 비중을 높입니다.
- 최초 결과는 `5장`을 보여줍니다.
- 최초 화면에는 `재생성` 버튼과 `미용사 상담` 버튼을 함께 둡니다.
- 재생성 후에는 `재생성` 버튼 없이 `미용사 상담` 버튼만 남깁니다.

백엔드 준비 상태:
- 공개 재생성 endpoint는 준비되어 있습니다.
- `recommendation_id` 기반 호출은 준비되어 있습니다.
- 재생성 성공 응답 형식은 준비되어 있습니다.

프론트 확인 체크리스트:
- [x] 최초 추천 결과 화면에서 `재생성` 버튼과 `미용사 상담` 버튼을 함께 배치했는지 확인
- [x] 재생성 완료 후 `재생성` 버튼을 숨기고 `미용사 상담` 버튼만 남기도록 처리했는지 확인
- [x] `recommendation_id`를 프론트 상태에서 안정적으로 보관하는지 확인
- [x] 동일 카드에 대해 재생성을 `1회만` 허용하도록 제어하는지 확인
- [x] 재생성 `대기 / 실패 / 완료` 상태를 어떤 UI로 보여줄지 확정했는지 확인
- [x] 재생성 완료 후 카드 갱신을 `전체 재조회`로 할지 `부분 갱신`으로 할지 확정했는지 확인

왜 필요한가:
- PM이 제품 방향은 정해줬고, 프론트에서는 버튼 노출 규칙과 상태 처리 방식을 구현하면 됩니다.

## `#140` 공통 에러 응답 표준화

PM 결정:
- 현재 권장 기본안으로 진행합니다.
- 즉, 당장은 DRF 기본 에러 응답을 우선 사용하고, 공통 envelope는 단계적으로 정리합니다.

백엔드 준비 상태:
- 현재 기본 에러 응답은 DRF `detail` 기준입니다.
- 반복되는 에러 응답을 helper로 정리할 준비는 되어 있습니다.

프론트 확인 체크리스트:
- [x] 우선 DRF 기본 에러 형식(`detail`, validation error)으로 연결 가능한지 확인
- [x] 프론트 에러 처리 코드가 공통 envelope 없이도 동작하는지 확인
- [x] 나중에 공통 에러 표준으로 바꿀 때 꼭 필요한 UI 필드가 무엇인지 정리했는지 확인

왜 필요한가:
- 지금은 에러 구조를 억지로 고정하지 않는 것이 더 안전합니다.
