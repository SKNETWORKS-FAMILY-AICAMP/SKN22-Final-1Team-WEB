# 백엔드 후속 요청 및 체크리스트 - 프론트 공유용

작성일: 0331
버전: ver1

## 1. 목적

현재 backend는 프론트와의 주요 연동 사항을 대부분 반영한 상태입니다.
이번 재검토에서는 `제3자 정보동의` 체크박스 생성 여부와, 회원가입 완료 후 대시보드 즉시 이동 흐름을 다시 확인했습니다.

이 문서는 프론트에서 바로 확인해야 할 사항과, 최근 backend pull 이후 보고된 오류의 원인 및 해결 방향을 한 문서에 정리한 것입니다.

## 2. 현재 backend 기준 반영 상태

- `/partner/`와 `/partner/login/`은 로그인 입구 화면, `/partner/dashboard/`는 실제 대시보드 화면으로 구분됩니다.
- 회원가입 성공 후 `/partner/dashboard/` 이동 기준 반영
- 관리자 세션 / 디자이너 세션 분리
- 관리자/디자이너 모두 챗봇 접근 허용
- 설문 `q1~q6` 저장 매핑 반영
- `detail`, `message`, `error_code`, `errors` 응답 형식 반영
- `storage_snapshot` 등 추적용 응답 필드 반영

즉, 현재 남은 핵심은 프론트 화면에서 입력, 에러 표시, redirect 흐름을 실제로 맞추는 일입니다.

## 3. 최근 오류 분석

### 3-1. 파트너 회원가입 400

최근 보고된 회원가입 400은 한 가지 원인으로 발생한 것이 아닙니다.
직접 검증 결과 아래 3가지 원인이 겹쳐 있습니다.

1. `agree_third_party_sharing` 체크박스 생성 확인
- backend는 `agree_terms`, `agree_privacy`, `agree_third_party_sharing`를 모두 필수로 요구합니다.
- 현재 프론트 회원가입 화면에 `agree_third_party_sharing` 체크박스가 실제로 생성되어 있는지 다시 확인해야 합니다.
- 이 항목이 화면에 정상 생성되어 있으면 더 이상 누락 원인으로 보지 않아도 됩니다.

2. 전화번호 중복
- 현재 입력값으로 사용된 `010-3746-6933`은 실제 DB에 이미 등록된 값입니다.
- 따라서 제3자 제공 동의값을 채워도, 다음에는 전화번호 중복으로 400이 발생합니다.

3. 사업자등록번호 유효성 실패
- 전화번호를 새 값으로 바꾸더라도, 현재 사용한 사업자등록번호 값은 backend 유효성 검사를 통과하지 못합니다.
- 따라서 사업자등록번호도 별도 오류 원인으로 존재합니다.

### 3-2. 해석

즉, 현재 프론트에서 보이는 400은 “사업자번호 하나만 틀려서 나는 오류”가 아니라,
아래가 순서대로 겹친 복합 오류입니다.

- 프론트/백엔드 계약 불일치
- 기존 데이터 중복
- 입력값 자체 유효성 실패

## 4. 해결 방향

### 4-1. 정책 확인이 필요한 부분

- `agree_third_party_sharing`가 실제로 필수라면:
  - 프론트에 체크박스와 문구를 추가해야 합니다.
- 필수가 아니라면:
  - backend에서 해당 필드를 optional로 완화해야 합니다.

### 4-2. 프론트에서 바로 확인하거나 보완할 부분

- 사업자등록번호가 잘못된 형식일 때 제출 전에 안내하거나 막는지 확인
- 이미 가입된 전화번호 입력 시 필드 단위 에러가 자연스럽게 보이는지 확인
- 회원가입 성공 후 `/partner/dashboard/`로 바로 이동하는지 확인
- 챗봇이 관리자 세션 / 디자이너 세션에서 바로 동작하는지 확인
- 설문 `q1~q6` 응답이 정상 전송되는지 확인
- backend가 내려주는 `errors` 응답을 화면에서 자연스럽게 보여주는지 확인

## 5. 체크리스트

### 5-1. 파트너 회원가입

- [ ] 회원가입 화면의 필수 동의 항목 구성이 실제 정책과 맞는지 확인
- [ ] `agree_third_party_sharing` 체크박스와 문구가 실제 화면에 생성되어 있는지 확인
- [ ] 사업자등록번호가 잘못된 형식일 때 제출 전에 안내하거나 막는지 확인
- [ ] 이미 가입된 전화번호 입력 시 `errors.phone`이 자연스럽게 보이는지 확인
- [ ] 사업자등록번호 오류 시 `errors.business_number`가 자연스럽게 보이는지 확인
- [ ] 파트너 가입 후 이동 경로가 `/partner/`인지 `/partner/dashboard/`인지 정책과 실제 화면이 일치하는지 확인
- [ ] 회원가입 성공 후 `/partner/dashboard/`로 바로 이동하는지 확인

설명:

회원가입 400은 여러 조건이 겹쳐 있는 상태입니다.
또한 파트너 가입 후 이동 경로는 현재 문서와 실제 화면에서 `/partner/`와 `/partner/dashboard/`가 혼재되어 있으므로, 어떤 화면을 최종 기준으로 둘지 먼저 맞춰야 합니다.
프론트에서는 어떤 값이 실제로 먼저 걸리는지 화면에서 구분 가능하도록 보여주는 것이 중요합니다.

### 5-2. 파트너 로그인 및 챗봇

- [ ] 관리자 세션에서 로그인 직후 챗봇을 바로 사용할 수 있는지 확인
- [ ] 디자이너 세션에서도 챗봇을 사용할 수 있는지 확인
- [ ] 세션 기반 접근 시 별도 페이지 이동 없이 동작하는지 확인

설명:

backend는 관리자/디자이너 세션 모두 챗봇 접근을 허용하고 있습니다.
프론트에서는 실제 대시보드 화면에서 이 흐름이 막히지 않는지만 확인하면 됩니다.

### 5-3. 설문 및 응답 처리

- [ ] 성별 설문 `q1~q6` 응답이 현재 화면에서 정상 전송되는지 확인
- [ ] backend가 내려주는 `detail`, `message`, `error_code`, `errors`를 화면이 무리 없이 처리하는지 확인
- [ ] `storage_snapshot` 같은 추가 응답 필드가 화면을 깨지 않는지 확인

설명:

backend는 기존 저장 구조와 호환되도록 설문 매핑과 응답 형식을 보강한 상태입니다.
프론트에서는 화면 표시와 사용자 흐름이 깨지지 않는지만 보면 됩니다.

## 6. 정리

현재 backend 기준 blocker는 대부분 해소된 상태입니다.
지금부터 중요한 것은 프론트 화면에서

- 입력 검증
- 필드별 에러 표시
- redirect
- 세션 기반 챗봇 접근

이 네 가지가 실제로 자연스럽게 이어지는지 확인하는 것입니다.

## 7. 0331 추가 피드백

- [ ] 회원가입 400 발생 시 상단 요약 문구가 "입력한 정보를 다시 확인해 주세요."로만 고정되지 않는지 확인
- [ ] `data.errors`의 첫 번째 실제 메시지(예: 중복 전화번호, 필수 동의 누락, 사업자등록번호 오류)가 화면에 그대로 보이는지 확인
- [ ] 회원가입 완료 후 실제 화면이 `/partner/dashboard/`로 즉시 이동하는지 확인

해설:

backend는 이미 `errors.phone`, `errors.business_number`, `errors.agree_third_party_sharing`처럼 구체적인 원인을 내려주고 있습니다.
그런데 화면에서 계속 일반 문구만 보이면 사용자가 어떤 값을 고쳐야 하는지 알 수 없습니다.
따라서 회원가입 오류 화면은 "실패했다"는 사실보다 "왜 실패했는지"를 먼저 보여주도록 확인이 필요합니다.

추가로, 회원가입 성공 시에는 `/partner/dashboard/`로 즉시 이동하는 흐름이 실제로 동작해야 합니다.
