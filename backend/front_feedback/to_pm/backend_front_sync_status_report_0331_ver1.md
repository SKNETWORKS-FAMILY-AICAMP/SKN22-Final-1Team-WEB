# 백엔드-프론트 동기화 상태 보고

작성일: 0331
버전: ver1

## 1. 현재 판단

현재 backend는 프론트와의 주요 동기화 작업을 대부분 반영한 상태입니다.
이번 재검토에서는 `제3자 정보동의` 체크박스 생성 여부와, 회원가입 완료 후 대시보드 즉시 이동 흐름을 다시 확인했습니다.

핵심적으로 확인된 내용은 아래와 같습니다.

- `/partner/`와 `/partner/login/`은 로그인 입구 화면입니다.
- `/partner/dashboard/`는 실제 파트너 센터 본문 화면입니다.
- 관리자 회원가입 후 `/partner/dashboard/` 이동 기준은 backend에 반영되어 있습니다.
- 관리자/디자이너 세션 분리 및 챗봇 접근은 backend에 반영되어 있습니다.
- 설문 `q1~q6` 저장 매핑과 `errors` 응답 형식도 backend에 반영되어 있습니다.
- `agree_third_party_sharing` 체크박스는 프론트 화면에 생성된 상태로 확인되었습니다.
- 회원가입 성공 후 `/partner/dashboard/`로 즉시 이동하는 흐름도 확인되었습니다.

## 2. 회원가입 400 원인 확정

직접 검증한 결과, 현재 400을 만드는 원인은 최소 2개입니다.

### 2-1. 전화번호 중복

현재 입력에 사용된 `010-3746-6933`은 실제 DB에 이미 등록되어 있습니다.

- `name`: 이병재
- `store_name`: mirrAI
- `phone`: `01037466933`
- `business_number`: `9800456120`

따라서 이 번호로는 다시 가입 시도 시 전화번호 중복으로 400이 발생합니다.

### 2-2. 사업자등록번호 유효성 실패

전화번호를 새 값으로 바꾸더라도, 현재 사용한 사업자등록번호 값은 backend 유효성 검사를 통과하지 못합니다.
즉 사업자등록번호도 별도 오류 원인으로 존재합니다.

## 3. 현재 backend에서 이미 반영된 사항

- 샵(`AdminAccount`) / 디자이너(`Designer`) / 고객(`Client`) 구조 반영
- 관리자 세션 / 디자이너 세션 분리
- 관리자/디자이너 모두 챗봇 사용 가능
- 회원가입 후 `/partner/dashboard/` redirect
- `q1~q6` 설문 저장 매핑
- `detail`, `message`, `error_code`, `errors` 응답 형식
- `agree_third_party_sharing` 체크박스 반영
- 회원가입 성공 후 대시보드 즉시 이동

## 4. 프론트와 함께 정리해야 하는 부분

### 4-1. 사업자등록번호 입력 UX

사업자등록번호 형식 검증은 backend에 이미 있으므로, 프론트에서도 자리수와 형식 안내를 먼저 넣는 편이 좋습니다.

### 4-2. 중복 데이터 안내

이미 가입된 번호로 다시 가입을 시도할 수 있으므로, 프론트에서는 `errors.phone`, `errors.business_number`를 사용자에게 자연스럽게 보여줄 필요가 있습니다.

### 4-3. 이동 경로 문서 정리

현재 문서와 실제 화면에서 `/partner/`와 `/partner/dashboard/`가 혼재되어 있으므로, 최종 기준을 문서에도 일치시켜 둘 필요가 있습니다.

## 5. 결론

현재 backend blocker는 대부분 해소된 상태입니다.
다만 파트너 회원가입 400은 여전히 전화번호 중복과 사업자등록번호 유효성 실패가 함께 존재하는 복합 오류이므로, 프론트와 backend의 입력/에러 표시 방식을 계속 맞춰가야 합니다.

정리하면:

- backend는 주요 기능과 응답 형식을 이미 반영한 상태
- `제3자 정보동의` 체크박스는 화면에 생성되었고 확인되었다
- 회원가입 성공 후 `/partner/dashboard/` 즉시 이동도 확인되었다
- 남은 오류 원인은 전화번호 중복과 사업자등록번호 유효성 실패
