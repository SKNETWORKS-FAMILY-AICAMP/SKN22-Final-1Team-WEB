# PM 후속 확인 요청 `ver1`

이 문서는 프론트 문서와 같은 이슈를 다루지만, 초점은 다릅니다.
여기서는 화면 구현이 아니라 정책, 범위, 우선순위, 전환 시점을 확인하려고 합니다.

## `#32` 매장 계정 생성

결정이 필요한 것:
- 가입 성공 직후 사용자를 어느 첫 화면으로 보낼지
- 대시보드 직행인지, 별도 온보딩 화면이 필요한지

지금 필요한 이유:
- 백엔드의 계정 생성은 준비됐지만, 가입 직후 흐름은 아직 확정되지 않았습니다.

권장 기본안:
- 별도 온보딩 요구가 없다면 매장 대시보드로 바로 보내는 흐름이 가장 단순합니다.

## `#123` 추천 시뮬레이션 재생성 API 제공

결정이 필요한 것:
- 이 기능을 사용자에게 얼마나 노출할지
- 재생성 횟수 제한이 필요한지
- 실패 시 어느 수준까지 안내할지

지금 필요한 이유:
- 백엔드 API는 준비됐지만, 제품 노출 수준은 아직 열려 있습니다.

권장 기본안:
- 우선 제한적으로 노출하고, 실패 안내는 명확히 두되 무제한 재시도는 피하는 쪽이 안전합니다.

## `#140` 공통 에러 응답 표준화

결정이 필요한 것:
- 공통 에러 응답 표준화를 이번 범위에 포함할지
- 단계적 전환인지, 일괄 전환인지
- 내부 디버그 정보와 사용자 노출 문구를 어디까지 분리할지

지금 필요한 이유:
- 지금도 개발은 진행 가능하지만, 표준화 시점이 늦어질수록 전환 비용이 커집니다.

권장 기본안:
- 당장은 현행 DRF 스타일을 유지하고, 별도 전환 이슈로 단계적으로 표준화하는 편이 안전합니다.

## `#141` 관리자 및 클라이언트 인증 갱신 정책 정리

결정이 필요한 것:
- refresh token이 이번 릴리즈에 필수인지
- 보안 우선인지, 세션 편의 우선인지
- refresh 미도입 시 허용 가능한 재로그인 부담이 어느 정도인지

지금 필요한 이유:
- 프론트 세션 UX와 백엔드 인증 계약이 같은 결정에 묶여 있습니다.

권장 기본안:
- 이번 범위에서는 access token 중심으로 가고, refresh는 명시적 필요가 생길 때 도입하는 편이 단순합니다.

## `#135` 모델 entrypoint, endpoint 추가

결정이 필요한 것:
- README와 실제 배포본 중 무엇을 기준 계약으로 볼지
- 배포된 handler 확인과 재배포 책임을 누가 맡을지
- 15:30 이후 모델 런타임 안정화 작업을 이번 범위에 포함할지

지금 필요한 이유:
- 백엔드 연결은 이미 되지만, 배포된 RunPod 동작이 문서와 완전히 일치하지는 않습니다.

권장 기본안:
- 우선 실제 배포본을 현재 사실 기준으로 보고, 이후 README를 맞추거나 재배포로 계약을 정렬하는 편이 현실적입니다.
